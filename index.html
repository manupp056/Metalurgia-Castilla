Metalurgia Castilla

Mantenimiento predictivo con AWS para una planta industrial en Burgos.

El problema

Metalurgia Castilla tiene 40 máquinas industriales en su planta de Burgos: hornos, prensas, tornos y cintas transportadoras. Hoy el mantenimiento se apoya en una única persona y en ficheros locales, sin ningún tipo de respaldo ni visibilidad para dirección.

Situación de partida

1 revisión manual por semana
(Excel + ficheros de texto)
0 copias de seguridad
(todo en un único PC de planta)
0 dashboards para dirección

Consecuencias

  • Detección tardía de fallos: si nadie revisa a tiempo, el fallo se detecta cuando ya ha ocurrido.
  • Riesgo de pérdida total del histórico: si el PC de planta se rompe, no queda ningún dato.
  • Sin datos (MTBF, MTTR, paradas) para justificar ninguna decisión de inversión ante dirección.
40 máquinas — Hornos (8) · Prensas (10) · Tornos (14) · Cintas transportadoras (8)

Plan de mantenimiento

Un modelo combinado, con umbrales propios por máquina y un flujo de trabajo claro.

Modelo combinado

Predictivo

Base del plan: sensores miden variables clave en tiempo real y se comparan contra umbrales para anticipar el fallo antes de que ocurra.

Preventivo

Para piezas que no se pueden sensorizar: revisión por calendario, con periodicidad fija.

Correctivo

Último recurso: reparación una vez que el fallo ya se ha producido.

Umbrales en 3 niveles

Cada máquina tiene sus propios umbrales Normal / Aviso / Crítico, calculados a partir del histórico de esa máquina en concreto — no un valor genérico igual para todas las máquinas del mismo tipo.

Dos hornos pueden tener un comportamiento distinto por antigüedad, uso o desgaste, así que compararlos contra el mismo número fijo daría falsas alarmas en uno y avisos tardíos en el otro. Por eso el umbral se afina con el propio histórico de cada máquina.

Por qué dos VPCs separadas

Antes había una sola VPC. Ahora el pipeline industrial (ingesta y procesamiento) está separado de las aplicaciones que usan las personas (dashboards, alarmas):

  • Un fallo o cambio en Athena o QuickSight no puede afectar a la recepción de lecturas de planta.
  • Cada VPC tiene su propio control de tráfico (Security Groups): aísla por red, no solo por permisos IAM. Así, si alguien compromete la VPC de aplicaciones, no llega directamente al pipeline de ingesta.
  • El VPC Peering solo permite el tráfico estrictamente necesario (Athena leyendo S3 y el Data Catalog de Glue), nada más.

Flujo end-to-end

Máquinas → Switch industrial (OT) → Edge Gateway (buffer + filtrado, MQTT local) → Customer Gateway → Site-to-Site VPN (IPsec) → VPN Gateway (VPC Gestión) → AWS IoT Core (IoT Rule) → AWS Lambda (evalúa umbrales) → Amazon S3 (guarda la lectura) + Amazon SNS si aviso/crítico (cruza por VPC Peering) → AWS Glue (crawler + ETL) → VPC Peering → Amazon Athena (consulta SQL vía Data Catalog) → Amazon QuickSight (dashboard).

Qué hace cada elemento

Hornos (8)

Qué hace: funden/tratan metal y generan lecturas de temperatura y consumo eléctrico.

Por qué está ahí: es la máquina de criticidad más alta del plan — un fallo aquí es el más caro de detectar tarde.

Se conecta con: el switch industrial (OT).

Prensas (10)

Qué hace: generan lecturas de vibración y consumo eléctrico.

Por qué está ahí: criticidad media del plan de mantenimiento.

Se conecta con: el switch industrial (OT).

Tornos (14)

Qué hace: generan lecturas de vibración y consumo eléctrico.

Por qué está ahí: criticidad baja, pero son el grupo más numeroso (14 máquinas).

Se conecta con: el switch industrial (OT).

Cintas transportadoras (8)

Qué hace: generan lecturas de consumo eléctrico.

Por qué está ahí: mantenimiento de soporte/preventivo, no predictivo intensivo.

Se conecta con: el switch industrial (OT).

Switch industrial (OT)

Qué hace: conecta las 40 máquinas dentro de la red OT, IP 192.168.10.2.

Por qué está ahí: sin salida a Internet — mantiene la red de planta aislada del resto.

Se conecta con: las 40 máquinas y el Edge Gateway.

Edge Gateway (EG)

Qué hace: Servidor Gestor Industrial, IP fija 192.168.10.10. Hace de buffer local y filtrado, y reenvía por MQTT/TLS.

Por qué está ahí: sigue funcionando aunque se caiga Internet, sin perder lecturas.

Se conecta con: el switch industrial y el Customer Gateway.

Customer Gateway

Qué hace: es el extremo VPN en el firewall/router de la planta.

Por qué está ahí: inicia el túnel cifrado hacia AWS.

Se conecta con: el Edge Gateway y, vía Site-to-Site VPN, con el VPN Gateway en AWS.

Site-to-Site VPN

Qué hace: túnel IPsec (2 túneles, IKEv2 + AES-256-GCM) entre la planta y AWS.

Por qué está ahí: cifra el tráfico planta↔AWS para que nadie pueda leerlo ni alterarlo por el camino.

Se conecta con: el Customer Gateway y el VPN Gateway.

VPN Gateway (VGW)

Qué hace: termina el Site-to-Site VPN en la subred pública 10.20.0.0/24.

Por qué está ahí: es la puerta de entrada del tráfico de planta a la VPC de Gestión.

Se conecta con: el Site-to-Site VPN y, dentro de la VPC, con AWS IoT Core.

AWS IoT Core

Qué hace: recibe y autentica cada lectura MQTT con un certificado X.509 por sensor/gateway.

Por qué está ahí: es el punto de entrada seguro de los datos al pipeline en AWS.

Se conecta con: el VPN Gateway y, vía IoT Rule, con AWS Lambda.

AWS Lambda

Qué hace: evalúa cada lectura contra los umbrales normal/aviso/crítico.

Por qué está ahí: es la lógica de decisión del sistema, sin servidor que mantener.

Se conecta con: IoT Core, Amazon S3 y Amazon SNS.

Amazon S3

Qué hace: data lake — guarda el histórico raw y processed.

Por qué está ahí: resuelve el riesgo de partida: "si el PC de planta se rompe, se pierde el histórico".

Se conecta con: Lambda (escritura), Glue (lectura) y AWS Backup.

AWS Glue

Qué hace: crawler + ETL que transforma los datos a Parquet y alimenta el Data Catalog.

Por qué está ahí: deja los datos listos y catalogados para consultarlos con SQL.

Se conecta con: Amazon S3 y, vía VPC Peering, con Amazon Athena.

AWS IAM

Qué hace: gestiona roles de mínimo privilegio para cada servicio.

Por qué está ahí: Lambda, Glue o QuickSight solo pueden hacer lo estrictamente necesario, sin credenciales compartidas.

Se conecta con: todos los servicios de la VPC de Gestión.

AWS KMS

Qué hace: gestiona las claves de cifrado SSE-KMS de S3.

Por qué está ahí: cifra el histórico en reposo — aunque alguien accediera al bucket, no podría leer los datos sin la clave.

Se conecta con: Amazon S3.

Amazon CloudWatch

Qué hace: vigila el propio pipeline (errores de Lambda, latencia de IoT Core).

Por qué está ahí: detecta problemas del sistema, no solo de las máquinas.

Se conecta con: IoT Core y Lambda.

AWS Backup

Qué hace: copias automatizadas de S3.

Por qué está ahí: es la respuesta directa al problema de origen del proyecto: cero copias de seguridad.

Se conecta con: Amazon S3.

Amazon Athena

Qué hace: consultas SQL serverless sobre S3 a través del Data Catalog de Glue.

Por qué está ahí: permite consultar el histórico sin montar ni mantener una base de datos.

Se conecta con: Glue (vía VPC Peering) y QuickSight.

Amazon QuickSight

Qué hace: dashboards para dirección e ingeniería.

Por qué está ahí: resuelve el problema de origen: cero dashboards para dirección.

Se conecta con: Amazon Athena.

Amazon SNS

Qué hace: distribuye alarmas por email y SMS al ingeniero de mantenimiento.

Por qué está ahí: convierte una lectura de aviso/crítico en una notificación inmediata.

Se conecta con: AWS Lambda, cruzando por VPC Peering hasta el ingeniero de mantenimiento.

VPC Peering

Qué hace: conecta la VPC de Gestión (10.20.0.0/16) con la VPC de Aplicaciones Corporativas (10.30.0.0/16).

Por qué está ahí: el tráfico va privado, nunca sale a Internet, y solo permite lo estrictamente necesario (Athena leyendo S3/Data Catalog).

Se conecta con: ambas VPCs.

Seguridad y costes

Cada medida resuelve un requisito de negocio concreto — nada está "porque sí".

Certificados X.509 por sensor/gateway

Autenticación individual en IoT Core: se puede revocar una sola máquina sin afectar a las otras 39.

TLS/MQTT + VPN IPsec (IKEv2 + AES-256-GCM)

Cifrado en tránsito planta↔AWS — nadie puede leer ni alterar las lecturas por el camino.

AWS IAM

Roles de mínimo privilegio: cada servicio (Lambda, Glue, QuickSight) solo puede hacer lo estrictamente necesario, sin credenciales compartidas.

AWS KMS

Cifrado en reposo (SSE-KMS en S3): aunque alguien accediera al bucket, no podría leer los datos sin la clave.

VPC de Gestión / VPC de Aplicaciones + Peering

Aislamiento por red entre ingesta y aplicaciones: reduce el radio de impacto de un fallo o brecha.

Amazon CloudWatch

Vigila el propio pipeline (errores de Lambda, latencia de IoT Core) — detecta problemas del sistema, no solo de las máquinas.

AWS Backup

Resuelve el riesgo de origen del proyecto: "si el PC de planta se rompe se pierde el histórico" — S3 + Backup lo protegen.

AWS Budgets

Alertas por umbral de gasto mensual — control de costes para un proyecto serverless de pago por uso.

Costes

  • Servicios serverless (IoT Core, Lambda, Athena, SNS) pagan por uso — el tráfico de 40 sensores es irregular.
  • Amazon S3 con ciclo de vida a Glacier tras 90 días.
  • Amazon QuickSight por sesión de usuario, no licencia fija.
  • AWS Backup con retención definida.
